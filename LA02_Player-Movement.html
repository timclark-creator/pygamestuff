Background knowledge

To move an object on screen, we track its position using variables and change those values each frame based on which keys are held down.

pygame.key.get_pressed() returns a list of booleans — one per key — so we can detect multiple keys held at once.

Success criteria

  • Use pygame.key.get_pressed() to detect held keys
  • Move a rectangle by changing x and y with a speed variable
  • Support both WASD and arrow keys

Python — raging_rectangles.py 
import pygame

pygame.init()
screen = pygame.display.set_mode((640, 480))
pygame.display.set_caption("Raging Rectangles")
clock = pygame.time.Clock()

# Player starting position and speed
player_x = 300
player_y = 220
player_w = 40
player_h = 40
player_speed = 5

running = True
while running:
    for event in pygame.event.get():
        if event.type == pygame.QUIT:
            running = False

    # Check which keys are currently held down
    keys = pygame.key.get_pressed()

    # Move UP
    if keys[pygame.K_w] or keys[pygame.K_UP]:
        player_y -= player_speed
    # Move DOWN
    if keys[pygame.K_s] or keys[pygame.K_DOWN]:
        player_y += player_speed
    # Move LEFT
    if keys[pygame.K_a] or keys[pygame.K_LEFT]:
        player_x -= player_speed
    # Move RIGHT
    if keys[pygame.K_d] or keys[pygame.K_RIGHT]:
        player_x += player_speed

    screen.fill((0, 0, 0))
    pygame.draw.rect(screen, (0, 0, 255), (player_x, player_y, player_w, player_h))
    pygame.display.flip()
    clock.tick(60)

Hints & key ideas

Why four separate if statements?
Using if/elif would only allow one direction per frame. Four separate if statements let all held keys fire simultaneously — so holding W and D together moves the player diagonally up-right.
Why does moving up subtract from y?
In pygame, y = 0 is at the top of the screen and increases downward. Subtracting from y moves the player toward the top of the screen (upward visually).
What does clock.tick(60) do?
It limits the loop to 60 frames per second. Without it the loop runs as fast as the CPU allows — movement speed would vary on different computers.

What's new at this level

Instead of separate player_x and player_y variables, we use a pygame.Rect to store position and size together. We update player_rect.x and player_rect.y directly and pass the Rect straight into pygame.draw.rect().

Key concept

pygame.Rect(x, y, width, height) stores all four values in one object and gives you free attributes like .centerx, .right, and .bottom — very useful for boundary checking and collision detection later.

Hints & key ideas

What does pygame.Rect(300, 220, 40, 40) store?
x = 300, y = 220, width = 40, height = 40. All four values in one object instead of four separate variables.
Starting in the bottom-right corner
The window is 640×480. The player is 40×40. So pygame.Rect(600, 440, 40, 40) places it flush in the bottom-right corner (640−40=600, 480−40=440).
Free attributes you get from a Rect
A pygame.Rect automatically gives you .centerx, .centery, .right, .bottom, .topleft and more — all calculated from x, y, width, height. Very useful in future activities.

import pygame

# Initialise pygame
pygame.init()

# Set up the display window
screen = pygame.display.set_mode((640, 480))
clock = pygame.time.Clock()

# Player setup — change speed here to affect the whole game
player_rect = pygame.Rect(300, 220, 40, 40)
player_speed = 8  # Increased for snappy feel

# Show speed in window title as a visual indicator
pygame.display.set_caption(f"Raging Rectangles — Speed: {player_speed}")

running = True
while running:
    # Handle window close event
    for event in pygame.event.get():
        if event.type == pygame.QUIT:
            running = False

    # Check held keys each frame
    keys = pygame.key.get_pressed()

    # ESC key quits the game cleanly
    if keys[pygame.K_ESCAPE]:
        running = False

    # Move the player rect
    if keys[pygame.K_w] or keys[pygame.K_UP]:
        player_rect.y -= player_speed
    if keys[pygame.K_s] or keys[pygame.K_DOWN]:
        player_rect.y += player_speed
    if keys[pygame.K_a] or keys[pygame.K_LEFT]:
        player_rect.x -= player_speed
    if keys[pygame.K_d] or keys[pygame.K_RIGHT]:
        player_rect.x += player_speed

    # Clear screen and draw player
    screen.fill((0, 0, 0))
    pygame.draw.rect(screen, (0, 0, 255), player_rect)
    pygame.display.flip()
    clock.tick(60)

# Clean exit
pygame.quit()

Hints & key ideas

How does the f-string show the speed in the title?
An f-string embeds a variable directly in a string: f"Speed: {player_speed}" becomes "Speed: 8" at runtime. Change the variable once and the caption updates automatically.
Single source of truth
If speed appeared as the literal number 8 in four different if blocks, you'd need to find and change it four times. One variable at the top means one change updates the whole program.
ESC via get_pressed() vs the event loop
Both work. get_pressed() detects a held key each frame — consistent with how movement is handled. pygame.KEYDOWN in the event loop detects a single press. For quitting, either is fine.

Challenge 2 — Exceptional

Add a second player (different colour) controlled by I J K L. When both shapes overlap using colliderect(), show "Collision!" in the window title. Every line or section must be commented.

Python — raging_rectangles.py 
import pygame

# Initialise pygame
pygame.init()

# Set up the display
screen = pygame.display.set_mode((640, 480))
clock = pygame.time.Clock()

# Player 1 — blue, controlled by WASD / arrow keys
p1_rect = pygame.Rect(200, 220, 40, 40)
p1_speed = 5
p1_colour = (0, 0, 255)

# Player 2 — red, controlled by IJKL
p2_rect = pygame.Rect(400, 220, 40, 40)
p2_speed = 5
p2_colour = (255, 50, 50)

running = True
while running:
    # Handle quit event
    for event in pygame.event.get():
        if event.type == pygame.QUIT:
            running = False

    # Get all currently held keys
    keys = pygame.key.get_pressed()

    # ESC to quit
    if keys[pygame.K_ESCAPE]:
        running = False

    # Player 1 movement (WASD / arrow keys)
    if keys[pygame.K_w] or keys[pygame.K_UP]:
        p1_rect.y -= p1_speed
    if keys[pygame.K_s] or keys[pygame.K_DOWN]:
        p1_rect.y += p1_speed
    if keys[pygame.K_a] or keys[pygame.K_LEFT]:
        p1_rect.x -= p1_speed
    if keys[pygame.K_d] or keys[pygame.K_RIGHT]:
        p1_rect.x += p1_speed

    # Player 2 movement (IJKL keys)
    if keys[pygame.K_i]:
        p2_rect.y -= p2_speed
    if keys[pygame.K_k]:
        p2_rect.y += p2_speed
    if keys[pygame.K_j]:
        p2_rect.x -= p2_speed
    if keys[pygame.K_l]:
        p2_rect.x += p2_speed

    # Check if the two players overlap using colliderect()
    if p1_rect.colliderect(p2_rect):
        pygame.display.set_caption("Raging Rectangles — Collision!")
    else:
        pygame.display.set_caption("Raging Rectangles")

    # Clear screen and draw both players
    screen.fill((0, 0, 0))
    pygame.draw.rect(screen, p1_colour, p1_rect)
    pygame.draw.rect(screen, p2_colour, p2_rect)
    pygame.display.flip()
    clock.tick(60)

# Clean exit
pygame.quit()

Hints & key ideas

How does colliderect() detect overlap?
It compares the edges of both rectangles. If p1's right edge passes p2's left edge AND p1's bottom passes p2's top (and vice-versa), they overlap. It returns True if so, False if not.
Why call set_caption() every frame?
Collision state changes every frame — players might overlap one frame and separate the next. Calling it inside the loop keeps the title in sync with the current state at all times.
Extension idea
Instead of updating the title, try drawing text on screen with pygame.font, or change the background colour to red on collision. Both are more visible than a title change and would make a great further extension.
